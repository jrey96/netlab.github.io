A VPN creates an encrypted tunnel through an untrusted network — typically the internet — so that traffic between two endpoints looks like gibberish to anyone watching in between. Understanding VPNs means understanding tunneling, encryption, and authentication.

// The Core Problem VPNs Solve

Imagine your company has offices in New York and London. Without a VPN, connecting them means sending sensitive data across the public internet — where any router along the path could potentially capture, read, or modify your packets. This is called a man-in-the-middle attack.

A VPN solves this with three things: Tunneling — wrapping your original packet inside a new one so the real destination is hidden. Encryption — scrambling the contents so even if intercepted, nothing is readable. Authentication — ensuring both ends are really who they say they are before any data flows.

The result: two private networks separated by the internet behave as if they're directly connected. Traffic that hits the VPN gateway is encrypted, tunneled across the internet, decrypted at the other end, and delivered — all transparently to the end user.

// Concept 01
Tunneling
Encapsulating one packet inside another. The inner packet is the original data. The outer packet has new headers for transport across the public network — hiding the real source and destination.
// Concept 02
Encryption
Scrambling data using a shared key so only the intended recipient can read it. AES-256 is the current gold standard — it would take longer than the age of the universe to brute-force.
// Concept 03
Authentication
Verifying both peers are legitimate before establishing the tunnel. Done via pre-shared keys (PSK), digital certificates (PKI), or username/password. Prevents rogue gateways.

// The Protocol Suite

IPsec
& IKE

IPsec is the industry-standard protocol suite for securing IP traffic. IKE (Internet Key Exchange) is the negotiation protocol that sets up IPsec. Together they handle authentication, key exchange, and encrypted data transport.

// IPsec Architecture

IPsec is not a single protocol — it's a suite of protocols that work together. At the highest level it has two modes and two protocols to choose from:

Transport Mode — only the payload is encrypted. The original IP header is kept intact. Used for end-to-end between two hosts (e.g., a laptop directly to a server). The real IP addresses are visible to network devices along the path.

Tunnel Mode — the entire original packet (header + payload) is encrypted and wrapped in a new IP header. Used for site-to-site VPNs between two gateways. The internal IP addresses are completely hidden from anyone on the internet.

AH — Authentication Header

Provides integrity and authentication — proves the packet wasn't tampered with and came from a legitimate sender. Uses HMAC (SHA-256/SHA-384).

⚠ AH does NOT encrypt the payload — anyone can still read the data. It only signs it. AH is rarely used alone in modern deployments.

ESP — Encapsulating Security Payload

Provides confidentiality, integrity, and authentication. Encrypts the payload using AES and adds an integrity check. This is what's used in nearly all VPN deployments.

✦ ESP in tunnel mode is the standard for site-to-site VPNs. Protocol number 50 in the IP header.

// IKE — Internet Key Exchange

Before two VPN gateways can exchange encrypted traffic, they need to agree on which algorithms to use and exchange encryption keys securely. This is IKE's job — it runs on UDP port 500.

IKE works in two phases. Phase 1 establishes a secure, authenticated channel between the two peers — this is called the ISAKMP SA (Security Association). Phase 2 uses that secure channel to negotiate the actual IPsec SAs that will protect the data traffic.

Modern deployments use IKEv2 — faster, more reliable, supports MOBIKE (surviving IP address changes, important for mobile VPN clients), and uses fewer round trips than IKEv1.

// IKE Phase 1
ISAKMP SA
Peers authenticate each other (PSK or certificates). Negotiate encryption (AES), hash (SHA), DH group, lifetime. Creates a secure management channel for Phase 2.
// IKE Phase 2
IPsec SA
Uses the Phase 1 channel to negotiate the actual data encryption parameters. Creates two unidirectional SAs (one each way). Sets lifetime and PFS (Perfect Forward Secrecy).
// Security Association
SA / SPI
A Security Association is a one-way agreement on all crypto parameters. Identified by an SPI (Security Parameter Index) — a 32-bit number in the ESP header that tells the receiver which SA to use.
